# Data Structure & 3rd-Party Usage

A core goal of Baumradar is to eliminate the tedious processing of raw Open Data for other developers. You can directly use the cleaned, processed SQLite databases generated by our backend in your own projects (e.g., iOS App, Web App, Data Analysis).

## Zero-Trust via Ed25519 Signatures

To give you absolute certainty that the data is authentic and hasn't been tampered with, every database is signed by the backend.
1. An **Ed25519 Public Key** is provided in the project repository (and often linked in `catalog.json`).
2. When downloading a file (e.g., `berlin.db.gz`), you also download its signature `berlin.db.gz.sig`.
3. Before opening the database, you should verify the file against the signature using the public key.

![Signature Verification](signature_verification.png)

*Note: If a file exceeds 50MB, it is split into chunks (`.001`, `.002`). You must concatenate these chunks at the binary level before verifying the overall signature and extracting the GZIP.*

## The SQLite Database

The extracted `.db` file contains two very simple, easy-to-understand tables:

### Table `trees`
Contains all individual trees.
- `id` (String): Unique ID of the tree.
- `latitude` (Double): WGS84 Latitude.
- `longitude` (Double): WGS84 Longitude.
- `genusDe` (String): Unified common name of the tree genus (e.g., "Birke" for Birch).
- `artDe` (String): Specific species (if known).

### Table `geofences`
Pre-calculated zones for grouped trees of the same species. Highly practical for route-collision detection, saving you from checking tens of thousands of individual trees.
- `id` (String): Unique zone ID.
- `latitude` (Double), `longitude` (Double): Center of the zone.
- `radiusMeters` (Int): Radius in meters (e.g., 50m for single trees, 100m for tree clusters).

## Usage
Start by fetching the entry point JSON (`catalog.json`). It contains the direct download links to the databases and signatures for all supported cities.
